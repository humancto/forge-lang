# Forge v0.7.0 — Expert Review Report

**Date:** 2026-04-12
**Baseline:** v0.7.0, 948 Rust tests passing in 4.14s, ~26k LOC
**Reviewers:** 5 parallel principal-level Rust expert agents
**Scope:** VM engine, DAP debugger, AOT compilation, test coverage, benchmark suite

---

## Executive Summary

Forge v0.7.0 marks a major milestone — the bytecode VM is now the default engine, with AOT compilation, a DAP debugger, test coverage, and cross-file LSP. All 948 tests pass. However, expert review identified **2 critical issues** that should be resolved before declaring v0.7.0 production-ready:

1. **Undefined behavior** in the VM's instruction dispatch (invalid opcode transmute)
2. **Use-after-free risk** from incomplete GC root scanning

Neither has caused failures in testing (the 948-test suite exercises correct bytecode only), but both are latent bugs that will manifest under adversarial or edge-case conditions.

Overall architectural quality is strong. The register-based VM design, mark-sweep GC, DAP protocol integration, and AOT C code generation are all well-structured. The issues below are fixable without architectural changes.

## 1. VM Engine (Default Execution Engine)

**Files:** `src/vm/machine.rs`, `src/vm/compiler.rs`, `src/vm/bytecode.rs`, `src/vm/gc.rs`, `src/vm/value.rs`
**Verdict:** CONCERN — fix S1 + C1 to reach APPROVE

### Critical

**S1. Undefined Behavior: `OpCode` transmute from unchecked `u8`**

- **Location:** `src/vm/machine.rs:1061`
- **Code:** `let opcode: OpCode = unsafe { std::mem::transmute(op) };`
- **Problem:** `OpCode` has ~43 variants (0..42). Any `op > 42` (from bytecode corruption, malformed `.fgb`/`.fgc` file, or serialization bug) produces genuine undefined behavior per the Rust spec. The `_ =>` catch-all arm in the dispatch match cannot save this because the transmute has already created an invalid enum discriminant.
- **Also affects:** `src/vm/jit/type_analysis.rs:60`, `src/vm/jit/ir_builder.rs:88`
- **Fix:** Replace with `TryFrom<u8>` conversion or an explicit `match op { 0 => OpCode::LoadConst, ... _ => return Err(...) }` table.
- **Risk if unfixed:** Memory corruption, arbitrary behavior on malformed bytecode input.

**C1. GC Root Scanning Misses Three Value Tables**

- **Location:** `src/vm/machine.rs:1940-1957`
- **Problem:** The GC root collection scans registers, globals, and frame closures — but `method_tables`, `static_methods`, and `struct_defaults` all hold `Value::Obj(GcRef)` values that are NOT scanned. If these are the only remaining references to their objects, the GC will free them, leaving dangling `GcRef` pointers. Subsequent method calls on those structs trigger use-after-free.
- **Fix:** Add `method_tables`, `static_methods`, and `struct_defaults` to the root scanning loop.
- **Risk if unfixed:** Use-after-free, silent data corruption, crashes under GC pressure.

### High

**S2. `unsafe impl Send for SendableVM` — Unenforced Invariant**

- **Location:** `src/vm/machine.rs:15`
- **Problem:** The `Send` impl assumes `jit_cache` is empty (no raw `*const u8` pointers). Nothing enforces this invariant — `fork_for_spawn()` creates a fresh VM, but direct `SendableVM(vm)` construction on a non-fresh VM would be unsound.
- **Fix:** Add `debug_assert!(child.jit_cache.is_empty())` in `fork_for_spawn()`, or make the constructor validate.

**S3. `mem::forget(jit)` — Unbounded Memory Leak**

- **Location:** `src/vm/machine.rs:1994`
- **Problem:** When a hot function is JIT-compiled, the `JitCompiler` is leaked via `mem::forget` to keep compiled code alive. Each leak is hundreds of KB (Cranelift module allocations). Under sustained execution with many distinct hot functions, memory grows without bound.
- **Fix:** Store compiled modules in a managed collection instead of leaking.

**P1. 65,536 Values Pre-allocated on Every VM Instantiation**

- **Location:** `src/vm/machine.rs:295`
- **Problem:** `registers: vec![Value::Null; MAX_REGISTERS]` allocates ~1MB+ on every `VM::new()`, including for spawned tasks, module imports, and trivial scripts.
- **Fix:** Grow the register file on demand, or reduce `MAX_REGISTERS` with a check-and-grow strategy.

**P2. `Arc<Chunk>` Clone in Hot Dispatch Loop**

- **Location:** `src/vm/machine.rs:1025`
- **Problem:** `c.function.chunk.clone()` executes an atomic reference count increment on every single instruction fetch. In tight loops executing millions of instructions, this is measurable overhead.
- **Fix:** Hoist the `chunk` reference outside the loop, re-acquiring only on frame changes (Call, Return).

### Medium

**C2. Register Overflow Wrapping**

- **Location:** `src/vm/compiler.rs:97-104`
- **Problem:** `next_register` is `u8` with no bounds check. Functions needing >255 registers wrap to 0 silently, corrupting earlier local variables.
- **Fix:** Return `Result` or panic with a clear "too many registers" message.

**C3. `GetField`/`SetField` Limited to 8-bit Constant Index**

- **Location:** `src/vm/machine.rs:1382, 1498`
- **Problem:** The `c` field in ABC encoding is 8 bits, limiting programs to 255 distinct field name constants. Larger programs will silently use the wrong constant.
- **Fix:** Use ABx encoding for field access opcodes when the constant pool exceeds 255 entries.

**R2. `len()` Returns Bytes, Not Characters**

- **Location:** `src/vm/machine.rs:1585`, `src/vm/builtins.rs:415`
- **Problem:** `s.len()` returns byte count, not character count. For non-ASCII strings, this differs from the interpreter (which likely uses character count). For an "internet-native" language, byte length is almost certainly wrong.
- **Fix:** Use `s.chars().count()` or document byte semantics explicitly.

**R3. Object Equality Always Returns False**

- **Location:** `src/vm/value.rs:268-276`
- **Problem:** `GcObject::equals` only covers `String` and `Array`, not `Object`. `{a: 1} == {a: 1}` returns `false` in the VM. The interpreter likely supports structural object equality.
- **Fix:** Add `Object` variant to `GcObject::equals` with recursive field comparison.

### Low

**C4. `And`/`Or` Evaluate Both Sides Eagerly**

- **Location:** `src/vm/machine.rs:1156-1165`
- **Problem:** No short-circuit evaluation. Side effects in the right operand execute even when the left operand determines the result. Parity issue if the interpreter short-circuits.

**C5. Negative Array Index Not Handled**

- **Location:** `src/vm/machine.rs:1526-1530`
- **Problem:** `*i as usize` wraps negative integers to large values, producing "index out of bounds" instead of "negative index" error.

**R1. `OpCode::Pop` Defined But Never Emitted or Handled**

- **Location:** `src/vm/bytecode.rs:52`
- **Problem:** The opcode exists but has no handler in the dispatch loop. Any emission would hit the unknown opcode error arm.

**P3. `Value::clone()` in Hot Arithmetic Paths**

- **Location:** `src/vm/machine.rs:1083-1086`
- **Problem:** Clones exist to work around the borrow checker (`&mut self` + register reads). For small types this is negligible, but prevents borrowing optimizations.

**P4. String-Based Native Function Dispatch**

- **Location:** `src/vm/machine.rs:2071`
- **Problem:** Native function name is cloned (`String`) on every call. An interned index would be zero-cost.

## 2. DAP Debugger

**Files:** `src/dap/mod.rs`, `src/interpreter/mod.rs` (debug integration)
**Verdict:** CONCERN — fix Issue 1 before shipping

### High

**D1. Mixed Stdin Lock Scopes Corrupt DAP Message Reads**

- **Location:** `src/dap/mod.rs:20, 35, 39`
- **Problem:** The outer loop holds a `BufReader` via `stdin.lock().lines()`. Inside the loop, `io::stdin().read_line()` and `io::stdin().lock().read_exact()` acquire separate locks, bypassing the outer BufReader's internal buffer. When a DAP client pipelines messages (which VS Code does: `initialize` + `setBreakpoints` + `configurationDone` + `launch` sent rapidly), buffered bytes are read from the wrong position, corrupting the message stream.
- **Fix:** Use a single `BufReader<Stdin>` for the entire session. Read headers and body through the same reader.
- **Risk if unfixed:** DAP session corruption under normal VS Code usage.

### Medium

**D2. `Content-Length` Parse Failure Desynchronizes Stream**

- **Location:** `src/dap/mod.rs:31`
- **Problem:** `unwrap_or(0)` on parse failure reads 0 bytes, fails to parse JSON, and continues — but the actual message body is still in the buffer. All subsequent reads are shifted.

**D3. Breakpoints Not Keyed by Source File**

- **Location:** `src/dap/mod.rs:136-152`
- **Problem:** `setBreakpoints` clears ALL breakpoints on every call regardless of `args["source"]`. For single-file debugging this is fine, but multi-file debugging (via imports) loses breakpoints from earlier files.

**D4. Stop Reason Read from Mutable Shared State After Change**

- **Location:** `src/dap/mod.rs:537-548`
- **Problem:** After the interpreter pauses, the DAP thread reads `action` to determine stop reason — but the DAP thread may have already changed `action` via a `resume_interpreter` call. The stop reason should be sent through the channel alongside the line number.

### Low

**D5. Unknown Commands Return `success: true`**

- **Location:** `src/dap/mod.rs:353`
- **Problem:** The catch-all handler returns an empty success response. DAP spec expects either a proper result or `success: false`.

**D6. Missing `exited` Event Before `terminated`**

- **Location:** `src/dap/mod.rs:454`
- **Problem:** DAP spec requires an `exited` event (with exit code) before `terminated`. Some clients may not clean up properly.

**D7. Stop-on-Entry Timeout Hard-Coded to 5 Seconds**

- **Location:** `src/dap/mod.rs:460`
- **Problem:** If the program takes >5s to reach its first statement (slow imports), the initial stopped event is silently dropped.

**D8. Three Mutex Locks Per Statement in Debug Mode**

- **Location:** `src/interpreter/mod.rs:1649-1651`
- **Problem:** `debug_check()` acquires `breakpoints`, `action`, and `step_depth` locks on every statement. For `Continue` with no breakpoints, this is pure overhead. Use `AtomicU8` for the action and an `AtomicBool` "has_breakpoints" fast-path.

## 3. AOT Compilation

**Files:** `src/native.rs`, `src/main.rs` (Build command)
**Verdict:** CONCERN — no showstoppers, medium-priority cleanup

### Medium

**A1. Hardcoded `/tmp` in Generated C Code**

- **Location:** `src/native.rs:146, 266`
- **Problem:** Generated C uses `/tmp/forge-native-XXXXXX` and `/tmp/forge-aot-XXXXXX` rather than `getenv("TMPDIR")`. On Linux with hardened `/tmp` (`noexec` mount), the temp file may not be usable. On macOS, the canonical temp directory is `/private/var/folders/...`.
- **Fix:** Use `getenv("TMPDIR")` with `/tmp` fallback in generated C.

**A2. Duplicate Code: Two Nearly Identical Build Functions**

- **Location:** `src/native.rs:7-59` and `src/native.rs:61-110`
- **Problem:** `build_native_launcher` and `build_native_aot` are ~95% identical. Same duplication in generated C templates (lines 126-366).
- **Fix:** Extract shared helper for the build process and a parameterized C template.

### Low

**A3. Cryptic Error When `cc` Is Missing**

- **Location:** `src/native.rs:41-47, 92-98`
- **Problem:** Error says "failed to invoke C compiler" but doesn't suggest `xcode-select --install` (macOS) or `apt install build-essential` (Linux).

**A4. Missing Edge Case Tests**

- **Location:** `src/native.rs` tests section
- **Problem:** No test for `c_string_escape` edge cases (null bytes, non-ASCII), no end-to-end bytecode round-trip test, no test for large bytecode arrays.

**A5. Bytecode Is Unprotected in AOT Binaries**

- **Location:** CLI output at `src/main.rs:880`
- **Problem:** "No source exposure" is accurate but bytecode is extractable with `strings`. Should be documented as a limitation, not a security guarantee.

## 4. Test Coverage

**Files:** `src/testing/mod.rs`, `src/interpreter/mod.rs` (coverage tracking)
**Verdict:** CONCERN — accuracy bugs produce incorrect percentages

### High

**V1. Top-Level Statements Not Tracked**

- **Location:** `src/interpreter/mod.rs:656-676`
- **Problem:** The `run()` method iterates `program.statements` directly and calls `exec_stmt()`, but never inserts into the coverage `HashSet`. Coverage insertion only happens in `exec_stmts()` (line 1603-1606). All top-level code (variable declarations, imports, setup) is invisible to coverage, producing **systematically deflated percentages**.
- **Fix:** Add coverage tracking in `run()`:
  ```rust
  if let Some(ref mut cov) = self.coverage {
      if spanned.line > 0 { cov.insert(spanned.line); }
  }
  ```

### Medium

**V2. Executable Line Heuristic Misclassifies Several Line Types**

- **Location:** `src/testing/mod.rs:297-325`
- **Problem:** The heuristic counts these as executable, but the interpreter never records them:
  - `} else {` and `} else if ... {` — parser assigns the `If` line, not the `else` line
  - `else {` on its own line — same reason
  - Opening braces `{` on their own line — only `}` is excluded
  - Decorator lines (`@test`, `@skip`) — metadata consumed by parser
  - `fn name(...) {` for non-test functions — goes through untracked `run()`
- **Effect:** Inflated denominator → deflated coverage percentages.

### Low

**V3. Block Comment Edge Case**

- **Location:** `src/testing/mod.rs:313-317`
- **Problem:** `  /* comment */ code` is incorrectly excluded because `trimmed.starts_with("/*")` is true and the `continue` skips the line despite containing executable code after the closing `*/`.

**V4. Coverage Is Interpreter-Only, Not Documented**

- **Problem:** `forge test --coverage` always uses the interpreter. If VM execution paths differ, coverage numbers may not reflect default-mode behavior. No warning is shown.

## 5. Benchmark Suite

**Files:** `benchmarks/run_benchmarks.sh`, `benchmarks/bench_*.fg`, `benchmarks/bench_*.py`
**Verdict:** NOT PUBLISHABLE — development tool only, cannot defend landing page claims

### High

**B1. External Timing Includes Process Startup**

- **Location:** `benchmarks/run_benchmarks.sh:38-42`
- **Problem:** Times are measured by spawning `python3 -c "import time; ..."` before/after each subprocess. This includes lexing, parsing, compilation, and VM initialization in the Forge measurement, and CPython startup in the Python measurement. For a 10ms JIT claim, the 30-80ms timing noise exceeds the signal.
- **Fix:** Add internal timing inside `.fg` and `.py` files, or use `hyperfine --warmup 3 --min-runs 10`.

**B2. Array Benchmark Is Not Apples-to-Apples**

- **Location:** `benchmarks/bench_array.fg` vs `benchmarks/bench_array.py`
- **Problem:** Forge builds arrays with a `while` loop calling `push()` 100,000 times. Python uses `list(range(100000))`, a single C-level call. This measures array construction speed (where Python wins by orders of magnitude), not map/filter/reduce performance.
- **Fix:** Use an explicit loop in Python: `arr = []; for i in range(100000): arr.append(i)`.

### Medium

**B3. Landing Page Numbers Are Unverifiable**

- **Location:** `docs/index.html:1488-1536`
- **Problem:** The landing page claims Rust 1.5ms, Go 4.2ms, Node.js 9.5ms, Forge JIT 10ms, Python 114ms on fib(30). The benchmark suite only tests Forge (VM mode) vs Python. No `bench_fib.rs`, `bench_fib.go`, or `bench_fib.js` files exist. The Rust/Go/Node.js numbers have no provenance.
- **Fix:** Add benchmark files for all claimed languages, or add a footnote with methodology and hardware specs.

**B4. Missing Engine Comparison**

- **Problem:** The script runs `forge run` (VM default) only. The landing page prominently features all three engines (JIT: 10ms, VM: 252ms, Interp: 2.3s), but the benchmark cannot reproduce these numbers.
- **Fix:** Run each benchmark with `forge run`, `forge run --jit`, `forge run --interp`.

**B5. No Statistical Rigor**

- **Problem:** Single run per benchmark, no warmup, no standard deviation, no machine specification beyond `uname -ms`. For casual development use this is fine. For landing page claims comparing against compiled languages, this is insufficient.

## Positive Findings

Across all five reviews, the experts noted several strengths:

- **Architecture is sound.** The register-based VM design, mark-sweep GC, DAP protocol integration, and AOT C code generation are all well-structured and maintainable.
- **Test coverage is excellent.** 948 tests with a 4-way parity harness (interpreter, VM, bytecode round-trip, JIT) provides strong regression protection.
- **Code organization is clean.** Clear module boundaries, sensible file sizes, good separation of concerns.
- **Generated C is correct.** The `mkstemp` + `fdopen` pattern avoids TOCTOU races, signal handling follows Unix conventions, arg array construction is correct.
- **DAP Condvar pattern is safe.** The flag-based resume mechanism compensates for lost notifications — a subtle correctness property that was verified.
- **Coverage performance is negligible.** One `if let` + `HashSet::insert` per statement adds unmeasurable overhead.

---

## Conclusion

Forge v0.7.0 is a substantial, well-engineered release. The VM's promotion to default engine is architecturally justified — it passes all 948 tests and supports the vast majority of Forge features. The two critical issues (S1: opcode transmute UB, C1: GC root gap) are narrow, well-understood bugs with straightforward fixes. Resolving Phase A items (~4 hours of work) would move all five subsystems to APPROVE status.

The benchmark suite needs the most work before landing page performance claims can be considered defensible, but this is a documentation/marketing concern rather than a correctness one.

**Recommendation:** Fix Phase A items, cut a v0.7.1 patch, then address Phase B items in the next development cycle.
